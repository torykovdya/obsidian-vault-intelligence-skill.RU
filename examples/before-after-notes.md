## Пример 1: Fleeting → Permanent

### ДО (Fleeting Note)
```
GPT-4o намного быстрее чем GPT-4. Надо использовать его везде.
```

### ПОСЛЕ (Permanent Note)
```markdown
# Latency vs Quality: когда скорость модели важнее точности

GPT-4o быстрее GPT-4 при сопоставимом качестве на большинстве задач.
Это меняет архитектурное решение: для задач реального времени (чатботы,
интерактивные агенты) скорость — это фича, которая важнее нескольких
процентов точности.

Правило: сначала пробуй более быструю модель. Если качество неприемлемо — апгрейд.
Не наоборот.

Связи: [[Model Selection Decision Tree]] | [[Agent UX Principles]]
Теги: #when-choosing-model #for-agent-architecture #latency-tradeoff
Источник: личный опыт, 2024-01
```

**Что изменилось:** наблюдение → принцип с правилом принятия решений

## Пример 2: Literature Note плохого качества → хорошая

### ДО (копипаст из статьи)
```
"The key to effective prompting is to think of the model as a 
very smart but very literal interpreter of instructions. 
It will do exactly what you say, not what you mean."
— из какой-то статьи
```

### ПОСЛЕ (Literature Note правильно)
```markdown
# LLM как буквальный интерпретатор, а не телепат

LLM выполняет то, что написано, а не то, что имеется в виду.
Это фундаментальная ошибка большинства промптов: мы предполагаем,
что модель "поймёт контекст". Она не понимает — она следует инструкции.

Практическое следствие: каждый промпт нужно читать глазами
буквального исполнителя. "Сделай это лучше" — инструкция без смысла.
"Перепиши, сократив на 30% без потери ключевых аргументов" — точная инструкция.

Связи: [[Precision in Prompt Instructions]] | [[Why Vague Prompts Fail]]
Теги: #mental-model-for-prompting #explains-prompt-failures
Источник: [название статьи, автор, ссылка]
```

**Что изменилось:** цитата → своя мысль с практическим выводом

## Пример 3: Слишком длинная заметка → разбивка на атомарные

### ДО (монолит)
```markdown
# Всё что я знаю про RAG

RAG - это Retrieval Augmented Generation. Работает так: берёшь документы,
делишь на чанки, создаёшь embeddings, кладёшь в векторную БД. Потом при
запросе ищешь похожие чанки и подаёшь в промпт.

Проблемы: качество чанкинга влияет на всё. Если чанки плохие — retrieval
плохой. Есть разные стратегии: fixed-size, semantic, hierarchical.

Ещё проблема — hallucinations всё равно бывают даже с RAG. Надо делать
evals. Можно использовать LLM-as-judge для оценки качества ответов.

Инструменты: LlamaIndex, LangChain, Chroma, Pinecone, Weaviate.
LlamaIndex удобнее для RAG чем LangChain по моему опыту.
```

### ПОСЛЕ (4 атомарных заметки)

**Заметка 1:**
```markdown
# RAG: принцип работы

Retrieval Augmented Generation — подход, при котором LLM получает
релевантный контекст из внешней базы перед генерацией ответа.

Цикл: документы → чанки → embeddings → векторная БД → запрос →
поиск похожих чанков → чанки + вопрос в промпт → ответ.

Ключевое: модель не "знает" документы, она их "читает" каждый раз заново.

Связи: [[Embeddings Explained]] | [[Vector DB Comparison]]
Теги: #rag-fundamentals #for-explaining-rag-to-others
```

**Заметка 2:**
```markdown
# Chunking strategy определяет качество всего RAG-pipeline

Качество чанкинга — самый недооценённый параметр в RAG.
Плохие чанки → плохой retrieval → плохие ответы, даже с хорошей моделью.

Три основных подхода:
- Fixed-size: просто, но теряет смысловые границы
- Semantic: сложнее, но чанки содержат законченные мысли
- Hierarchical: документ → секции → параграфы, позволяет zooming

Правило: начинай с semantic chunking. Fixed-size только если скорость критична.

Связи: [[RAG Pipeline Architecture]] | [[Semantic Splitting Libraries]]
Теги: #rag-deep-dive #chunking #when-building-rag
```

**Заметка 3:**
```markdown
# RAG не устраняет галлюцинации — только снижает их

Распространённое заблуждение: RAG = нет галлюцинаций.
Реальность: RAG снижает вероятность, но модель всё ещё может
генерировать несуществующие детали поверх реального контекста.

Решение: evals. Конкретно — LLM-as-judge: другая модель проверяет,
поддерживается ли ответ контекстом.

Связи: [[LLM-as-Judge Pattern]] | [[Evals Pipeline Design]]
Теги: #rag-limitations #hallucinations #when-building-production-rag
```

**Заметка 4:**
```markdown
# LlamaIndex vs LangChain для RAG: личный опыт

LlamaIndex специализируется на data ingestion и retrieval.
LangChain универсален, но RAG — не его главная сила.

На практике: LlamaIndex требует меньше boilerplate для типичных RAG-задач.
LangChain лучше, если RAG — часть сложного агентного пайплайна.

Источник: личный опыт при построении [проект]
Связи: [[AI Framework Selection]] | [[RAG Pipeline Architecture]]
Теги: #framework-comparison #rag-tooling
```

**Что изменилось:** 1 монолит → 4 заметки, каждую можно найти и использовать независимо

## Пример 4: Orphan Note → интегрированная

### ДО (orphan, без единой ссылки)
```markdown
# Второй мозг

Концепция второго мозга — перенести часть когнитивной нагрузки
на внешнюю систему. Книга Тьяго Форте.
```

### ПОСЛЕ
```markdown
# Второй мозг как внешняя когнитивная система

Тьяго Форте: "второй мозг" — внешняя система для хранения и
обработки информации, снижающая когнитивную нагрузку.

Ключевой принцип: мозг создан для генерации идей, не для их хранения.
Хранение — работа системы.

Разница с Zettelkasten: PARA-метод Форте ориентирован на проекты,
Zettelkasten — на знания. Можно совмещать: PARA как структура папок,
Zettelkasten как качество заметок.

Связи: [[Zettelkasten vs PARA]] | [[Cognitive Offloading]] | [[PKM Comparison]]
Теги: #pkm-foundations #for-explaining-note-systems #when-choosing-vault-structure
Источник: Building a Second Brain — Тьяго Форте
```

**Что изменилось:** изолированный факт → узел с 3 связями и ясным контекстом

## Пример 5: Плохие теги → умные теги

### ДО
```
Tags: #ai #tools #useful #productivity #notes
```

### ПОСЛЕ
```
Tags:
#when-evaluating-new-ai-tools        ← контекст поиска
#for-article-comparing-llm-ides      ← будущее использование
#contradicts-my-earlier-take-on-X    ← интеллектуальное напряжение
#requires-follow-up-research         ← незакрытая петля
```

**Правило:** хорошие теги — это маленькие вопросы или ситуации, а не категории.
